to UNHEX/HEX in every query, so the helper functions earn their keep.

PROMPT 6:
[continue — services layer]

AI RESPONSE SUMMARY:
Three services landed:
- RandomInviteCodeService implementing InviteCodeService — a SecureRandom-backed generator
  with a 5-attempt retry loop on collision (calls existsByCode each round). Configurable
  length and alphabet via @Value-injected properties. Throws IllegalStateException if all
  attempts collide.
- InMemoryLobbyRegistry implementing LobbyRegistry — a ConcurrentHashMap of `code -> Lobby`,
  where each Lobby has its own ReentrantLock plus two maps (sid -> Player, lower-cased
  nickname -> sid). The lock makes the "check capacity, then insert" check atomic so two
  threads can't both squeeze into the last slot. The join() method also handles reattach: if
  a player rejoins with the same nickname after a disconnect, the registry swaps the STOMP
  session id without emitting a duplicate "player_joined".
- GameSessionService — orchestrates createSession / startSession / cancelSession / kickPlayer.
  Every host-only operation re-reads the underlying Game and verifies the caller owns it
  (HostMismatchException). Status guards reject non-LOBBY transitions where appropriate.
  Empty-lobby starts throw EmptyLobbyException (per spec edge case 2). Broadcasts go to BOTH
  /topic/host/{code} and /topic/game/{code} so host UI and player UI stay in sync.

WHAT I LEARNED:
ConcurrentHashMap alone isn't enough when you need to make a *compound* decision atomically
("is there room AND if so, add a row"). The per-lobby ReentrantLock is what actually prevents
the over-fill bug. Just using `synchronized(this)` would lock all lobbies against each other
which would tank throughput, so per-key locking is the right granularity.

wrong URL. Because lobby.js never loaded, the player browser never sent the
/app/lobby/{code}/join STOMP message, which is why the server saw 0 players. The AI fixed all
six templates (login, register, dashboard, join, host_lobby, player_lobby) to use @{/css/...}
and @{/js/...}, restarted the app, verified both assets returned 200, and confirmed roster
sync started working with two browser windows.

BUGS AND HOW I FIXED IT:
The auth pages had the same wrong URL prefix from before this PR but I never noticed because
they had no important client-side JavaScript — the page rendered fine without the stylesheet,
so I missed the silent 404. The new lobby pages NEEDED lobby.js to function, which made the
404 visible. Lesson: a 404 on a non-essential CSS file is invisible until something actually
depends on it.

WHAT I LEARNED:
Two things. One: Spring Boot's default static-resource path mapping is the URL root, not
/static. The convention is to write `<link href="/css/foo.css">`, not `<link
href="/static/css/foo.css">`. Two: when something looks like "the WebSocket isn't connecting",
check the simpler stuff first — is the JS even loading? The browser console told the truth
(404 on lobby.js); I didn't have to debug STOMP at all.

PROMPT 12:
"also, do you guys want to run on docker or locally?" says the other dev. what's the
implication for both?

AI RESPONSE SUMMARY:
The AI noted the repo already had a Dockerfile and docker-compose.yml so the question was
really about inner-loop dev speed. Local (`mvn spring-boot:run` + host MySQL): ~1s app
restarts, instant Thymeleaf reloads, working IDE breakpoints, lower CPU/RAM. But "works on my
machine" risk around JDK and MySQL versions. Full Docker (`docker compose up --build`):
identical Linux JDK + MySQL across the team, schema auto-loaded from the volume mount,
matches CI/prod. But ~30–60s per Java change rebuild vs ~1s locally, and remote-debug is
awkward. Recommended middle path: hybrid — DB in Docker (so teammates don't argue about MySQL
versions), app on host (so iteration stays fast). For the assignment scope I told the AI
local was fine, and it agreed.

WHAT I LEARNED:
The right answer for "Docker vs local" depends on what changes most often. For a team where
the DB schema changes rarely but the app code changes constantly, putting the DB in Docker and
the app local hits the sweet spot. Going all-in on Docker for dev would have made every code
change painful for a 30-second rebuild that I don't need at this stage of the project.

AI RESPONSE SUMMARY:

BUGS AND HOW I FIXED IT:
But the app's logs showed `Access denied for user 'quiz_user'@'172.18.0.4'` looping with
Hibernate "Unable to determine Dialect" errors stacking on top of it. Cause: my mysql_data
Docker volume was leftover from an earlier compose run that had pinned a different password.
MySQL only honors MYSQL_USER/MYSQL_PASSWORD on FIRST volume init — subsequent restarts ignore
those env vars even if you change .env. Fix: `docker compose down -v` to wipe the persistent
volume, then `docker compose up -d --build` to recreate from scratch. After the wipe, the
schema initialized correctly with the new password and the app booted. I verified all four
tables (users, games, questions, game_sessions) existed inside the container and /auth/register
rendered fine.

WHAT I LEARNED:
Docker named volumes are persistent across `down`/`up` cycles. That's normally a good thing
(your DB data isn't wiped every restart) but it bit me here because changing the password in
.env didn't actually reach MySQL — the user was already created in the volume from a previous
run with a different password. The `-v` flag on `docker compose down` removes the volumes
along with the containers, which is what you want when you intentionally want a clean slate.
This is worth adding to the setup guide's troubleshooting section so the next teammate
doesn't trip over it.
