SESSION 1: Backend Architecture Debugging & Layer Synchronization
================================================================================

PROMPT 1:
"location: class com.quiz.repository.QuestionRepository [ERROR] -> [Help 1]... cannot find symbol method findByGame(int)... location: variable questions of type com.quiz.repository.QuestionRepository... incompatible types: byte[] cannot be converted to java.util.UUID... im still encountering errors I have a bunch of errors"

AI RESPONSE SUMMARY:
- Diagnosed a fundamental "handshake" mismatch between the Service layer, the Data model, and the Repository layer. 
- Pointed out that `GameEngineService` was calling getter methods (like `getQuestionPrompt()` and `getChoiceA()`) that did not exist in the `Question.java` model.
- Identified that `QuestionRepository.java` was missing the core `findByGame(int)` fetch logic required to load quizzes into the engine.
- Corrected a data-type mismatch in `GameRepository.java` where a MySQL HEX string was incorrectly being parsed into a `java.util.UUID` object instead of the `byte[]` expected by the system's `Game.java` model.

WHAT I LEARNED:
- **Strict Contract Enforcement:** In strongly typed languages like Java, the MVC layers must have perfectly aligned contracts. A single missing method or mismatched field name will cause cascading Maven build failures.
- **Data Mapping Nuances:** Bridging relational database primitives (like MySQL's `BINARY(16)` or `HEX` representations) to Java backend types requires careful handling within Spring's `RowMapper`. 
- **Alias Methods:** I learned how to use alias getter methods in a model to satisfy different naming conventions required by different service layers without breaking existing code.

CODE SNIPPET USED:
```java
// Fixing the MySQL HEX to byte[] conversion in GameRepository
private static final RowMapper<Game> MAPPER = (rs, i) -> {
    Game g = new Game();
    g.setGameID(rs.getInt("gameID"));
    
    // Convert the HEX string from DB into the byte[] your Game model expects
    String hex = rs.getString("userID");
    if (hex != null) {
        g.setUserID(javax.xml.bind.DatatypeConverter.parseHexBinary(hex));
    }
    // ...
    return g;
};
```

ISSUES ENCOUNTERED/SOLVED:
- **The "String to Char" Error:** Even after fixing the names, the compiler threw `Character.toUpperCase(char) is not applicable (argument mismatch; java.lang.String cannot be converted to char)`. The AI helped me realize my alias `getCorrectAnswer()` returned a `String`, but the service expected a `char`. I solved this by redirecting the service to natively call `getCorrectOption()`.
- **Ghost Errors in VS Code:** Discovered that the Java Language Server sometimes gets stuck caching old errors. I learned to use `mvn clean compile` and "Clean Java Language Server Workspace" to force a fresh assessment.

SESSION 2: Full-Stack Feature Development - Manual Quiz Creation
================================================================================

PROMPT 2:
"I need to make a feature so that when someone is creating/hosting their own quiz they can input their own sets of questions manually with multiple choice set of answers (4 possible answers). Where would this fit in the new file structure and what files do you need"

AI RESPONSE SUMMARY:
- Proposed a comprehensive three-part MVC solution to integrate the feature without muddying existing code.
- Created `ManualGameController.java` to handle HTTP routing and insertion logic.
- Created `ManualGameRequest.java` as a Data Transfer Object (DTO) to elegantly capture the dynamic list of questions coming from the frontend.
- Built a Thymeleaf template (`create-manual.html`) featuring a dynamic form powered by JavaScript to let users add as many questions as they want.

WHAT I LEARNED:
- **Data Transfer Objects (DTOs):** I learned how DTOs act as a critical structural bridge, allowing complex HTML form payloads (like lists of nested objects) to be safely caught and mapped into Java classes before touching the database.
- **Spring @ModelAttribute Binding:** Discovered how Spring Boot magically binds indexed HTML input names (e.g., `name="questions[0].prompt"`) directly into a Java `List<QuestionInput>` within the DTO.
- **Code Reusability:** Rather than writing new SQL statements, I learned to leverage my previously debugged `GameRepository.insert()` and `QuestionRepository.insert()` methods, keeping the controller lean.

CODE SNIPPET USED:
```java
// Logic bridging the DTO payload to the Database Repositories
@PostMapping("/create")
public String processManualCreate(@ModelAttribute ManualGameRequest request, HttpSession session) {
    UUID userId = (UUID) session.getAttribute("userID");
    if (userId == null) return "redirect:/auth/login";

    // 1. Create the Game record first to retrieve the generated gameId
    int gameId = games.insert(userId, request.getTitle());

    // 2. Loop through the DTO's dynamic list and insert each question
    for (ManualGameRequest.QuestionInput input : request.getQuestions()) {
        Question q = new Question(gameId, input.getPrompt(), input.getCorrect(), 
                                  input.getA(), input.getB(), input.getC(), input.getD());
        questions.insert(q);
    }

    return "redirect:/dashboard"; // Send user back to see their newly created game
}
```

ISSUES ENCOUNTERED/SOLVED:
- **Orphaned Feature:** I successfully built and compiled the controller and views, but completely forgot to build a way for the user to actually navigate to the feature from the main application flow. This required a follow-up session to integrate it into the Dashboard UI.

SESSION 3: Dashboard UI/UX Integration & Flexbox Refactoring
================================================================================

PROMPT 3:
"ok i built it but on the website theres no actual option to use the feature i just made... Where is the section where i have my action buttons and i need to add the new button... regenerate this section... show before and after of where to place"

AI RESPONSE SUMMARY:
- Directed the exact placement of the new "Manual Creator" entry point inside the dashboard's `.actions-bar` container.
- Completely refactored the action cards section to utilize advanced CSS Flexbox properties to handle dynamically sized sibling elements.
- Applied unique styling (dashed borders, semantic background colors) to establish visual hierarchy and distinguish the "Manual Builder" conceptually from the "AI Generator".

WHAT I LEARNED:
- **Advanced Flexbox Mechanics:** By applying `align-items: stretch` to the parent container, I ensured that all action cards matched the height of the tallest card (the AI upload form), preventing an uneven, jagged layout.
- **Sticky Flex Children:** I learned how to use `margin-top: auto` on action buttons inside a flex-column card. This forces the button to stick to the absolute bottom of the card, ensuring alignment across all UI cards regardless of varying paragraph lengths above them.
- **Contextual UI Design:** Adding a feature isn't just about dropping a link on a page; it's about semantic placement. Grouping the manual creation next to the AI creation logically organizes the user's "Host" workflows.

CODE SNIPPET USED:
```html
<a class="action-card manual" th:href="@{/manual/create}" 
   style="display:flex; flex: 1; min-width: 300px; border: 2px dashed #CBD5E1; transition: transform 0.2s;">
  <div class="action-icon">✍️</div>
  <div class="action-body" style="display: flex; flex-direction: column; height: 100%;">
    <div class="action-title">Manual Creator</div>
    <div class="action-desc">Build your quiz from scratch. Perfect for custom exams or specific trivia.</div>
    <span class="action-btn" style="background: var(--navy); color: white; margin-top: auto;">Write Questions →</span>
  </div>
</a>
```

ISSUES ENCOUNTERED/SOLVED:
- **Wrapping & Overflow:** When the fourth card was added, the UI broke horizontally on smaller screens. Solved by introducing `flex-wrap: wrap;` and `flex: 1; min-width: 300px;` allowing the grid to dynamically reflow from 4-columns to 2-columns depending on viewport width.

REFLECTION:
This development sprint highlighted the critical difference between writing code that "works in isolation" versus writing code that "integrates cleanly." Initially, I struggled because my mental model of the backend was disjointed—I assumed if the repository was written, the service could just use it. Fighting through the Maven compilation errors forced me to deeply understand the strict contracts required between layers in Spring Boot. 

Furthermore, building the Manual Creation feature end-to-end (from the database layer all the way to the CSS flex properties on the Dashboard) gave me a profound appreciation for full-stack cohesiveness. I realized that backend logic is essentially invisible if the frontend UI doesn't provide a seamless, intuitive gateway for the user. By the end of this sprint, resolving the complex flexbox layout issues felt just as rewarding as fixing the tricky `RowMapper` byte conversions, as both were required to deliver a polished final product.
